### Шаблон для менеджера

Заполни описание задачи так, чтобы разработчик и тестировщик поняли её одинаково.

```markdown
## Контекст

Зачем нужна задача?
Какая проблема есть сейчас?
Кого это затрагивает: пользователь, поддержка, бизнес, внутренняя команда?
Есть ли ссылки на макеты, логи, тикеты, аналитику, обсуждения?

## Ожидаемый результат

Что должно измениться после выполнения задачи?
Какое поведение считается правильным?
Что пользователь или система должны увидеть?

## Критерии приёмки

- [ ] Сценарий 1: ...
- [ ] Сценарий 2: ...
- [ ] Ошибочный сценарий: ...
- [ ] Граничный случай: ...
- [ ] Проверены логи/API/аналитика, если применимо.
- [ ] Обновлена документация, если применимо.
```

### Мини-чеклист ясности

- [ ] Не используется формулировка «как обсуждали» без краткого резюме.
- [ ] Не используется формулировка «сделать красиво» без конкретных признаков результата.
- [ ] Если написано «не работает», есть шаги воспроизведения.
- [ ] Есть ожидаемое поведение.
- [ ] Есть критерии, по которым QA сможет сказать: «готово» или «не готово».
