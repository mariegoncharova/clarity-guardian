# Communication Guide для Clarity Guardian

Clarity Guardian нужен не для бюрократии, а чтобы менеджер, разработчик и тестировщик не играли в испорченный телефон.

Хорошая задача отвечает на три вопроса:

1. Почему мы это делаем?
2. Что должно получиться?
3. Как понять, что задача готова?

---

## Минимальный формат задачи

```markdown
## Контекст

Опиши проблему и причину задачи.

Плохо:
Нужно пофиксить оплату.

Хорошо:
Пользователь не может оплатить заказ картой, если банк возвращает 3DS challenge.
Сейчас после возврата из 3DS форма остаётся в состоянии загрузки.
Проблема влияет на оплату заказов через web checkout.

## Ожидаемый результат

Опиши правильное поведение.

Пример:
После успешного прохождения 3DS пользователь возвращается на страницу оплаты.
Статус заказа обновляется на `paid`.
Пользователь видит экран успешной оплаты.

## Критерии приёмки

- [ ] Пользователь может пройти оплату с 3DS challenge.
- [ ] После возврата из 3DS статус заказа становится `paid`.
- [ ] При ошибке банка пользователь видит понятное сообщение.
- [ ] Повторная попытка оплаты доступна.
- [ ] В логах есть payment_id и bank_response_code.
```

---

## Запрещённые мутные формулировки

Лучше не писать:

- «сделать красиво»
- «как обсуждали»
- «как в прошлый раз»
- «срочно»
- «пофиксить»
- «не работает» без шагов воспроизведения

Почему плохо:

- разработчик будет додумывать;
- тестировщик будет тестировать не то;
- менеджер получит результат «формально готово, но не то».

---

## Как писать баг

```markdown
## Контекст

Пользователь не может применить промокод при оформлении заказа.
Проблема воспроизводится на production и testing.

## Шаги воспроизведения

1. Открыть страницу оформления заказа.
2. Ввести промокод `SALE10`.
3. Нажать «Применить».

## Фактический результат

Появляется ошибка `Something went wrong`.

## Ожидаемый результат

Промокод применяется.
Сумма заказа пересчитывается.
Пользователь видит скидку 10%.

## Критерии приёмки

- [ ] Валидный промокод применяется.
- [ ] Невалидный промокод показывает понятную ошибку.
- [ ] Истёкший промокод показывает отдельный текст ошибки.
- [ ] Сумма заказа пересчитывается без перезагрузки страницы.
```

## Как разработчику передавать задачу в тестирование

Перед меткой `ready-for-testing` добавь в PR или Issue:

```markdown
## Что проверять

- Основной сценарий оплаты картой.
- Возврат из 3DS.
- Ошибку банка.
- Повторную попытку оплаты.

## Как проверять

1. Создать заказ.
2. Выбрать оплату картой.
3. Использовать тестовую карту с 3DS.
4. Завершить challenge.
5. Проверить статус заказа.

## Ожидаемое поведение

- После успешного 3DS заказ становится `paid`.
- После ошибки банка пользователь видит понятный текст.
- Повторная попытка оплаты доступна.

## Риски

- Можно сломать оплату без 3DS.
- Можно сломать повторную оплату.
- Можно потерять обновление статуса заказа.
```

## Главное правило

Если важная информация живёт только в чате, значит её нет.

Переноси договорённости в задачу.
